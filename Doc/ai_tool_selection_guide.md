# AI開発ツール使い分けガイド

## 目的

Cvnet10プロジェクトでは、**GitHub Copilot** と **OpenCode** の2つのAIツールを併用します。
このガイドは、各ツールの特性を理解し、適切に使い分けるための指針を提供します。

---

## 基本方針

### GitHub Copilot（VS2026統合）
- **役割:** リアルタイムコード補完・クイックフィックス・小規模編集
- **利用場面:** コーディング中の継続的なサポート
- **設定ファイル:** `.github/copilot-instructions.md`、`.github/copilot/*.agent.md`

### OpenCode（CLI/エディタ）
- **役割:** 大規模機能実装・複数ファイル横断作業・ドキュメント作成
- **利用場面:** 計画的な機能追加・リファクタリング・調査分析
- **設定ファイル:** `AGENTS.md`

### 共通ルール
- どちらのツールも作業履歴は `Doc/aicording_log.md` に記録
- Layer構造とRead-Onlyプロジェクトのルールを厳守
- 日本語でのコメント・ドキュメント記述

## 判断フローチャート

```
タスク開始
    ↓
編集対象は何ファイル？
    ├─ 1-2ファイル → 修正内容は軽微？
    │               ├─ Yes → **Copilot** 👍
    │               └─ No  → 構造変更あり？
    │                        ├─ Yes → **OpenCode** 👍
    │                        └─ No  → **Copilot** 👍
    │
    └─ 3ファイル以上 → **OpenCode** 👍

ドキュメント作成？
    ├─ コメントのみ → **Copilot** 👍
    └─ Markdown/図 → **OpenCode** 👍

トラブル調査？
    ├─ 1箇所のエラー → **Copilot** 👍
    └─ 複数/根本原因 → **OpenCode** 👍

新規機能実装？
    ├─ メソッド単位 → **Copilot** 👍
    └─ 画面単位以上 → **OpenCode** 👍
```

## 実践例：シナリオ別の使い方

### シナリオ1: 新規マスタメンテ画面の追加

**要件:** 「商品カテゴリマスタ」のメンテ画面を追加したい

**推奨アプローチ:**

1. **OpenCodeで骨格作成（20分）**
   ```
   MasterShohinMenteView を参考に、
   01Master フォルダに MasterCategoryMenteView を作成してください。
   ```
   - View/ViewModel/MenuData を一括作成
   - BaseMenteViewModel 継承で共通機能を利用

2. **GitHub Copilotで詳細実装（30分）**
   - VS2026 で ViewModel を開く
   - カスタムバリデーションロジックを Copilot で補完
   - 検索条件の Where 句を Copilot で記述

3. **OpenCodeで最終確認（5分）**
   ```
   MasterCategoryMenteView の実装をレビューして、
   問題があれば指摘してください。
   ```

**合計所要時間:** 約55分

---

### シナリオ2: 重複コードの共通化

**要件:** ViewModels 配下で重複している初期化ロジックを共通化したい

**推奨アプローチ:**

1. **OpenCodeで調査（15分）**
   ```
   ViewModels/01Master/ 配下で重複しているコードパターンを
   特定してください。
   ```
   - 重複箇所を自動検出
   - 共通化可能な箇所をリスト化

2. **OpenCodeでリファクタリング（30分）**
   ```
   検出した重複ロジックを BaseMenteViewModel に移動してください。
   影響範囲を Doc/ に記録してください。
   ```
   - 複数ファイルを同時編集
   - Build確認まで自動実行

3. **GitHub Copilotで微調整（10分）**
   - VS2026 で個別のViewModelを開く
   - 共通化しきれなかった部分を手動調整

**合計所要時間:** 約55分

---

### シナリオ3: Build エラーの解決

**要件:** 依存関係変更後に複数のエラーが発生

**エラー内容:**
```
CS0246: The type or namespace name 'ObservableCollection' could not be found
CS1061: 'MasterTokuiMenteViewModel' does not contain a definition for 'InitCommand'
... (他10件)
```

**推奨アプローチ:**

1. **OpenCodeで根本原因を特定（10分）**
   ```
   以下のBuildエラーが出ています。根本原因を特定してください：
   [エラーログをペースト]
   ```
   - 複数エラーから共通原因を特定
   - 修正方針を提示

2. **OpenCodeで一括修正（15分）**
   ```
   提示された修正方針で修正を実行してください。
   ```
   - 複数ファイルを同時修正
   - Build確認

3. **残存エラーはGitHub Copilotで個別対応（5分）**
   - VS2026でエラー箇所を開く
   - Copilotのクイックフィックスで解決

**合計所要時間:** 約30分

## ツール選択の判断基準まとめ

### GitHub Copilot を選ぶべき場合
✅ 編集対象が **1-2ファイル** で軽微な変更
✅ **インライン補完** でスムーズに記述できる内容
✅ VS2026で **コーディング中** に思いついた改善
✅ **クイックフィックス** で即座に修正可能
✅ **既存パターンの踏襲** が明確

### OpenCode を選ぶべき場合
✅ 編集対象が **3ファイル以上** または複数プロジェクト横断
✅ **新規画面・機能** の一括作成
✅ **大規模リファクタリング** で影響範囲の管理が必要
✅ **ドキュメント作成・更新** が主目的
✅ **トラブルシューティング** で根本原因の特定が必要
✅ **計画的な作業** で事前に全体像を把握したい

## ベストプラクティス

### 1. 小→大の原則
小さな修正はCopilot、大きな変更はOpenCode。
迷ったら、まずCopilotで試して、複雑なら OpenCode へ切り替える。

### 2. 履歴の一元化
両ツールとも `Doc/aicording_log.md` に記録。
どちらのツールで何を実施したか、後から追跡可能にする。

### 3. 定期的な見直し
月1回、使い分けルールの効果を確認し、必要に応じて調整。

### 4. Read-Onlyルールの厳守
どちらのツールを使う場合でも、以下のプロジェクトは編集禁止：
- `CodeShare`
- `Cvnet10Asset`
- `Cvnet10Base`
- `Cvnet10BaseMariadb` / `Cvnet10BaseOracle` / `Cvnet10BaseSqlite`

### 5. Layer構造の理解
両ツールとも Layer 構造を理解しているが、最終確認は開発者が実施。
- Layer 0/1/1.2: 原則Read-Only
- Layer 1.5: ドメインロジック配置
- Layer 2: UI/gRPCサービス実装

## まとめ

### 効果的な使い分けのポイント

1. **日常的なコーディング → GitHub Copilot**
   - VS2026で開発中は常にCopilotを活用
   - 思考の流れを止めずに素早く実装

2. **計画的な機能追加 → OpenCode**
   - 新規画面や大規模変更はOpenCodeで計画
   - 全体を俯瞰してから実装開始

3. **両者の連携**
   - OpenCodeで骨格作成 → Copilotで詳細実装
   - Copilotで実装 → OpenCodeでレビュー・リファクタリング

4. **作業履歴の一元管理**
   - どちらのツールを使っても `Doc/aicording_log.md` に記録
   - プロジェクト全体の変更履歴を把握

### 使い分けの成功指標（KPI）

- **開発速度:** 導入前比 20-30% 向上を目標
- **コード品質:** レビュー指摘件数 20% 削減を目標
- **ドキュメント充実度:** ドキュメント更新頻度 50% 向上を目標
- **開発者満足度:** 主観評価で 4/5 以上を目標
